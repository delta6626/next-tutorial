// 01

What is Next.js?

Next.js is a React framework built by Vercel. Your React and TypeScript knowledge transfers completely — Next.js just adds structure on top.

React alone

  • Runs entirely in the browser
  • You configure routing yourself (React Router)
  • You set up bundling (Vite, CRA)
  • API calls go to a separate backend
  • SEO is difficult (no server rendering)

React + Next.js

  • Renders on server AND client
  • File-based routing built in
  • Zero-config bundling with Turbopack
  • API routes live in the same project
  • First-class SEO support
Key Insight

Next.js doesn't replace React — it's React with opinions. Every component you write is still a React component.

// 02

Project Setup

Creating a Next.js project is similar to create-react-app or npm create vite@latest:

terminal bash 
# Create a new project
npx create-next-app@latest my-app

# Prompts you'll see:
# ✔ Would you like to use TypeScript? → Yes
# ✔ Would you like to use ESLint?     → Yes
# ✔ Would you like to use Tailwind?   → Your choice
# ✔ Would you like to use App Router? → Yes (use this)
# ✔ Would you like to use Turbopack?  → Yes

cd my-app
npm run dev  # starts at http://localhost:3000
Tip

Always choose App Router (not Pages Router). It's the modern default since Next.js 13. This guide covers App Router only.

// 03

File Structure

The app/ directory is where everything lives. Unlike React, the file structure defines your routes.

my-app/ ├── app/ ← your app lives here │ ├── layout.tsx ← root layout (like index.html) │ ├── page.tsx ← the "/" route │ ├── globals.css │ ├── about/ │ │ └── page.tsx ← the "/about" route │ ├── blog/ │ │ ├── page.tsx ← "/blog" │ │ └── [slug]/ │ │ └── page.tsx ← "/blog/any-post" │ └── api/ │ └── users/ │ └── route.ts ← API endpoint ├── components/ ← shared components (your choice) ├── public/ ← static files └── next.config.ts
The Rule

A folder becomes a route only if it contains a page.tsx file. A folder without one is invisible to the router.

// 04

The App Router

Forget <Route path="..."> — in Next.js, routing is purely file-system based. Here's the mental model:

File pathURL
app/page.tsx/
app/about/page.tsx/about
app/blog/page.tsx/blog
app/blog/[slug]/page.tsx/blog/anything
app/shop/[...rest]/page.tsx/shop/a/b/c
app/api/users/route.tsAPI: /api/users

Special filenames that Next.js recognizes inside any route folder:

FilePurpose
page.tsxThe UI for this route (publicly accessible)
layout.tsxShared wrapper that persists across child routes
loading.tsxAutomatic loading skeleton (Suspense boundary)
error.tsxError boundary for this route segment
not-found.tsxCustom 404 for this segment
route.tsAPI endpoint (GET, POST, etc.) — no UI
// 05

Pages & Layouts

page.tsx — A route's UI

A page.tsx is just a React component exported as default:

app/about/page.tsx tsx 
// This is just a React component — no magic here
export default function AboutPage() {
  return (
    <main>
      <h1>About Us</h1>
      <p>Hello from /about</p>
    </main>
  )
}

layout.tsx — Persistent wrappers

Layouts wrap pages and don't re-render on navigation. Use them for nav bars, sidebars, and shared structure. The root layout is required and replaces your index.html:

app/layout.tsx tsx 
import type { Metadata } from 'next'

export const metadata: Metadata = {
  title: 'My App',
  description: 'My Next.js app',
}

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body>
        <nav>My Nav Bar</nav>
        {children}  {/* ← pages render here */}
      </body>
    </html>
  )
}
Tip

You can nest layouts. app/dashboard/layout.tsx wraps all dashboard routes without affecting the rest of your app.

// 06

Dynamic Routes

Square brackets create dynamic segments, just like React Router's :slug — but it's the folder name:

app/blog/[slug]/page.tsx tsx 
// TypeScript: params is async in Next.js 15+
export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params

  return <h1>Post: {slug}</h1>
}

// Visiting /blog/hello-world → slug = "hello-world"

Catch-all routes

app/docs/[...path]/page.tsx tsx 
// /docs/a/b/c → params.path = ['a', 'b', 'c']
export default async function DocsPage({
  params,
}: {
  params: Promise<{ path: string[] }>
}) {
  const { path } = await params
  return <div>Path: {path.join('/')}</div>
}
// 07

Navigation

Replace <a> tags and React Router's <Link> with Next.js's own <Link>:

components/Nav.tsx tsx 
import Link from 'next/link'

export function Nav() {
  return (
    <nav>
      <Link href="/">Home</Link>
      <Link href="/about">About</Link>
      <Link href="/blog">Blog</Link>
    </nav>
  )
}

Programmatic navigation

For button clicks and form submissions, use the useRouter hook — but only in Client Components (more on that soon):

components/BackButton.tsx tsx 
'use client'  // required for hooks

import { useRouter } from 'next/navigation'  // ← NOT 'next/router'

export function BackButton() {
  const router = useRouter()

  return (
    <button onClick={() => router.push('/dashboard')}>
      Go to Dashboard
    </button>
  )
}
Common mistake

Import from 'next/navigation', not 'next/router'. The old Pages Router used next/router — it won't work in the App Router.

// 08

Server Components

This is the biggest mental shift from plain React. In Next.js, every component is a Server Component by default.

Server Components run only on the server — they never ship JavaScript to the browser. This means you can:

app/page.tsx tsx 
// No 'use client' → this is a Server Component
// You can use async/await directly at the component level!

async function getUsers() {
  const res = await fetch('https://api.example.com/users')
  return res.json()
}

export default async function HomePage() {
  const users = await getUsers()  // ← fetch on the server!

  return (
    <ul>
      {users.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  )
}

// What you CAN'T do in a Server Component:
// ✗ useState, useEffect, useRef, or any hook
// ✗ Event handlers (onClick, onChange)
// ✗ Access browser APIs (window, document)
Why this matters

Server Components reduce your bundle size dramatically — database queries, heavy libraries, and API calls stay on the server. The browser only receives HTML.

// 09

Client Components

When you need interactivity — state, effects, event handlers — add 'use client' as the very first line of the file:

components/Counter.tsx tsx 
'use client'  // ← this makes it a Client Component

import { useState } from 'react'

export function Counter() {
  const [count, setCount] = useState(0)

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  )
}

// Client Components work exactly like your React components today.
// All hooks, event handlers, and browser APIs work normally.

Mixing them together

A Server Component can import and render a Client Component — this is the pattern you'll use everywhere:

app/page.tsx tsx 
import { Counter } from '@/components/Counter'  // Client

// This page itself is a Server Component
export default async function Page() {
  const data = await fetchSomeData()  // server-side

  return (
    <div>
      <h1>{data.title}</h1>
      <Counter />  {/* client component inside server component ✓ */}
    </div>
  )
}

// ⚠️ The reverse doesn't work:
// You CANNOT import a Server Component inside a Client Component.
// 10

When to Use Each

Need this?Use
Fetch data from a DB or APIServer Component (default)
Access environment secretsServer Component
Render large amounts of static HTMLServer Component
useState, useReducerClient Component ('use client')
useEffect, useLayoutEffectClient Component ('use client')
onClick, onChange, event handlersClient Component ('use client')
Browser APIs (window, localStorage)Client Component ('use client')
Third-party interactive librariesClient Component ('use client')
Strategy

Keep components as Server Components until you need interactivity. Push 'use client' as far down the tree as possible — ideally to small leaf components like buttons and inputs, not entire pages.

// 11

Data Fetching

No more useEffect + useState for initial data. In Server Components, fetch data directly:

app/products/page.tsx tsx 
type Product = {
  id: number
  name: string
  price: number
}

async function getProducts(): Promise<Product[]> {
  const res = await fetch('https://api.example.com/products', {
    next: { revalidate: 3600 }  // cache for 1 hour
  })

  if (!res.ok) throw new Error('Failed to fetch')
  return res.json()
}

export default async function ProductsPage() {
  const products = await getProducts()

  return (
    <div>
      {products.map((p) => (
        <div key={p.id}>{p.name} — ${p.price}</div>
      ))}
    </div>
  )
}

Caching options

fetch caching options tsx 
// Cache indefinitely (static — like getStaticProps)
fetch(url, { cache: 'force-cache' })

// Never cache (dynamic — like getServerSideProps)
fetch(url, { cache: 'no-store' })

// Revalidate every N seconds (ISR equivalent)
fetch(url, { next: { revalidate: 60 } })

For databases (Prisma, Drizzle, etc.)

app/users/page.tsx tsx 
import { db } from '@/lib/db'  // your Prisma client

export default async function UsersPage() {
  // Query the DB directly — this runs on the server
  const users = await db.user.findMany()

  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>
}
// 12

API Routes

Create route.ts files (not page.tsx) to build REST API endpoints inside your Next.js project. Export named functions for each HTTP method:

app/api/users/route.ts tsx 
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

// GET /api/users
export async function GET(request: NextRequest) {
  const users = [{ id: 1, name: 'Alice' }]
  return NextResponse.json(users)
}

// POST /api/users
export async function POST(request: NextRequest) {
  const body = await request.json()

  // create user in DB...
  return NextResponse.json(
    { message: 'User created', user: body },
    { status: 201 }
  )
}
Note

If your page components query your database directly (Server Components), you often don't need API routes at all. API routes are most useful for external clients, mobile apps, or webhooks.

// 13

Server Actions

Server Actions let you run server-side code from a form or button click — without writing an API route. They're the recommended way to handle mutations (create, update, delete).

app/contact/page.tsx tsx 
export default function ContactPage() {

  // This function runs ON THE SERVER when the form submits
  async function submitContact(formData: FormData) {
    'use server'  // ← marks this as a Server Action

    const email = formData.get('email') as string
    const message = formData.get('message') as string

    // save to DB, send email, etc.
    await saveToDatabase({ email, message })
  }

  return (
    <form action={submitContact}>  {/* ← pass the function directly */}
      <input name="email" type="email" />
      <textarea name="message" />
      <button type="submit">Send</button>
    </form>
  )
}

Server Actions in Client Components

Extract them into a separate file with 'use server' at the top:

app/actions.ts tsx 
'use server'

export async function deleteUser(id: number) {
  await db.user.delete({ where: { id } })
}
components/DeleteButton.tsx tsx 
'use client'

import { deleteUser } from '@/app/actions'

export function DeleteButton({ id }: { id: number }) {
  return (
    <button onClick={() => deleteUser(id)}>
      Delete
    </button>
  )
}

Metadata & SEO

No more <Helmet>. Export a metadata object or a generateMetadata function from any page or layout:

app/blog/[slug]/page.tsx tsx 
import type { Metadata } from 'next'

// Static metadata
export const metadata: Metadata = {
  title: 'My Blog',
  description: 'Articles about stuff',
}

// OR: Dynamic metadata based on route params
export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>
}): Promise<Metadata> {
  const { slug } = await params
  const post = await getPost(slug)

  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      images: [post.coverImage],
    },
  }
}

Images & Fonts

next/image

Use <Image> instead of <img> — it auto-optimizes, lazy loads, and prevents layout shift:

components/Avatar.tsx tsx 
import Image from 'next/image'

export function Avatar() {
  return (
    <Image
      src="/avatar.jpg"
      alt="Profile photo"
      width={64}
      height={64}
    />
  )
}

next/font

Load Google Fonts with zero layout shift and no external network requests at runtime:

app/layout.tsx tsx 
import { Inter } from 'next/font/google'

const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
})

export default function RootLayout({ children }) {
  return (
    <html className={inter.className}>
      <body>{children}</body>
    </html>
  )
}

Environment Variables

Next.js has a built-in convention for env vars that controls what's exposed to the browser:

.env.local bash 
# Server-only (never sent to browser)
DATABASE_URL=postgresql://...
API_SECRET_KEY=super-secret

# Exposed to the browser (prefix with NEXT_PUBLIC_)
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_ANALYTICS_ID=UA-12345
usage in code tsx 
// Server Component or API route — works fine
const db = process.env.DATABASE_URL

// Client Component — must use NEXT_PUBLIC_ prefix
const apiUrl = process.env.NEXT_PUBLIC_API_URL
Security

Never use DATABASE_URL or secret keys in Client Components. They'll be bundled into the client-side JavaScript and exposed to anyone.

Deployment

Vercel (easiest)

1

Push to GitHub

Commit your project and push to a GitHub, GitLab, or Bitbucket repository.

2

Import on Vercel

Go to vercel.com, click "Import Project", and select your repository. Zero config needed.

3

Add environment variables

In the Vercel dashboard, add your .env.local values under Settings → Environment Variables.

4

Deploy

Every push to main auto-deploys. Pull requests get preview URLs automatically.

Self-hosted (Node.js)

terminal bash 
npm run build   # creates .next/ folder
npm run start   # starts production server on port 3000
Also works

Next.js deploys to AWS, Google Cloud, Railway, Fly.io, Docker, and any Node.js host. Run npm run build && npm start and it just works.

What's next?

You now have the core mental models. Explore these topics as you build:

Middleware — run code before requests hit your routes (auth, redirects). Parallel Routes — render multiple pages in the same layout. Intercepting Routes — open modals with sharable URLs. Cache & revalidation — revalidatePath() and revalidateTag() for on-demand cache busting.
